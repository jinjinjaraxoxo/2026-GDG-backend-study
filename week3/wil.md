## 3주차 WIL

### 1. Repository 계층이란?

Repository는 데이터베이스에 접근하여 CRUD(Create, Read, Update, Delete)를 수행하는 계층이다.  

계층형 아키텍처에서 전체 흐름은 다음과 같다.

- Controller: 클라이언트의 요청을 받는다.
- Service: 비즈니스 로직을 처리한다.
- Repository: 데이터베이스에 접근한다.
- DB: 실제 데이터가 저장되는 공간이다.

즉, Service는 직접 DB에 접근하지 않고 Repository를 통해 데이터를 다룬다.

### 2. ORM과 JPA

기존에는 데이터베이스를 사용하려면 SQL을 직접 작성해야 했다.  
하지만 SQL을 직접 사용하는 방식은 반복적인 코드가 많고, 객체지향 언어인 Java와 관계형 데이터베이스 사이의 차이로 인해 개발이 번거롭다.

이 문제를 해결하기 위한 개념이 ORM(Object-Relational Mapping)이다.  
ORM은 객체와 데이터베이스 테이블을 자동으로 매핑해 주는 기술이다.

JPA(Java Persistence API)는 자바 진영에서 ORM을 사용하기 위한 표준 기술이다.  
JPA를 사용하면 SQL을 직접 작성하지 않고도 객체를 저장, 조회, 수정, 삭제할 수 있다.  
대표적인 JPA 구현체로는 Hibernate가 있다.

### 3. EntityManager와 영속성 컨텍스트

JPA는 EntityManager를 통해 엔티티를 관리한다.  
EntityManager는 엔티티를 영속성 컨텍스트(Persistence Context)라는 공간에 저장하고 관리한다.

영속성 컨텍스트는 JPA가 엔티티를 관리하기 위한 가상 공간이다.  
엔티티를 한 번 조회하면 이 공간에 저장되고, 같은 트랜잭션 안에서는 동일한 객체를 재사용한다.

예를 들어,

- 등록: `em.persist()`
- 조회: `em.find()`
- 삭제: `em.remove()`

와 같은 방식으로 동작한다.

### 4. 영속성 컨텍스트의 특징

영속성 컨텍스트에는 다음과 같은 특징이 있다.

#### 4-1. 1차 캐시

한 번 조회한 엔티티를 내부에 저장해두고 다시 재사용한다.  
따라서 같은 트랜잭션 안에서 같은 엔티티를 다시 조회할 때 DB에 재접근하지 않을 수 있다.

#### 4-2. 동일성 보장

같은 트랜잭션 안에서 같은 엔티티를 조회하면 항상 같은 객체를 반환한다.  
즉, 단순히 값이 같은 것이 아니라 참조 자체가 같은 객체임이 보장된다.

#### 4-3. 쓰기 지연

엔티티를 저장한다고 해서 즉시 DB에 INSERT SQL이 실행되는 것이 아니라,  
변경 내용을 모아두었다가 트랜잭션 커밋 시점에 한 번에 DB에 반영한다.

#### 4-4. 변경 감지(Dirty Checking)

엔티티를 조회한 후 값을 변경하면, JPA가 이를 자동으로 감지해서  
트랜잭션 커밋 시점에 UPDATE SQL을 실행한다.

즉, JPA에서는 수정용 SQL을 직접 작성하기보다  
조회한 엔티티의 값을 변경하는 방식으로 수정 작업을 처리할 수 있다.

### 5. 트랜잭션

트랜잭션은 여러 데이터베이스 작업을 하나의 논리적인 작업 단위로 묶는 개념이다.  
모든 작업이 성공하면 커밋(commit)되어 DB에 반영되고,  
중간에 하나라도 실패하면 롤백(rollback)되어 이전 상태로 되돌아간다.

예를 들어 주문 처리 과정에서

- 주문 정보 저장
- 상품 재고 차감
- 회원 포인트 적립

이 세 작업 중 하나라도 실패하면 전체 작업이 취소되어야 한다.  
이처럼 트랜잭션은 원자성(Atomicity)을 보장하기 위해 필요하다.

스프링에서는 보통 서비스 계층 메서드에 `@Transactional`을 붙여 트랜잭션을 관리한다.  
조회 전용 메서드에는 `@Transactional(readOnly = true)`를 적용할 수 있다.

### 6. Repository 구현 방식

이번 과제에서는 Member, Product, Order에 대한 Repository 계층을 구현했다.  
각 Repository는 `@Repository`를 사용해 스프링 빈으로 등록하고,  
`EntityManager`를 주입받아 CRUD 기능을 구현한다.

대표적으로 사용한 방식은 다음과 같다.

- 저장: `em.persist(entity)`
- 단건 조회: `em.find(Entity.class, id)`
- 목록 조회: JPQL 사용
- 삭제: `em.remove(entity)`

목록 조회나 특정 조건 조회에서는 JPQL을 사용한다.  
JPQL은 SQL과 비슷하지만 테이블이 아니라 엔티티 객체를 기준으로 작성한다는 차이가 있다.

예를 들어 회원 전체 조회는 다음과 같이 작성할 수 있다.

```java
em.createQuery("select m from Member m", Member.class).getResultList();
```

### 7. 느낀 점 

이번 주차는 이전보다 개념이 훨씬 어렵게 느껴졌다.
특히 영속성 컨텍스트, 트랜잭션, 변경 감지 같은 개념은 처음에는 추상적으로 느껴졌지만,
EntityManager의 동작 흐름과 함께 보니 왜 JPA가 편리한지 이해할 수 있었다.

기존에는 데이터 수정 시 직접 update 쿼리를 작성해야 한다고 생각했는데,
JPA에서는 조회한 엔티티의 값을 바꾸기만 해도 자동으로 반영된다는 점이 인상적이었다.
또한 단순히 CRUD 기능을 구현하는 것을 넘어서,
스프링 애플리케이션이 계층별로 어떤 역할을 분리해서 가지는지 더 명확하게 이해할 수 있었다.

다음 주차에는 엔티티와 데이터베이스 설계(ERD)를 배우게 되는데,
이번 주차에 배운 JPA 개념이 그 내용을 이해하는 데 중요한 기반이 될 것 같다.
